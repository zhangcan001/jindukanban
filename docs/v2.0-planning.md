# v2.0 下一代功能规划

## 1. 当前版本基础

当前稳定版本为 `v1.9-portable-release`，定位为本地单机便携版工程进度管理系统，已经具备从 Excel 导入到看板分析、预警、报表导出、备份诊断和 portable 发布包的完整闭环。

已具备的核心基础：

- Excel 导入与动态字段识别
- 项目模板和字段映射模板
- 进度看板、楼层统计、楼栋楼层统计
- 规则化进度分析说明
- 预警记录楼栋 / 楼层 / 系统 / 施工项增强
- 当前看板 Excel、Word 周报、滞后项整改清单导出
- 报表历史
- 本地备份、本地诊断和 portable 发布包

v1.9 的价值在于把“能导入、能统计、能导出、能迁移”做稳。v2.0 不应推翻这些基础，而应围绕真实工程管理中更高频的后续动作继续深化。

## 2. v2.0 总体目标

v2.0 建议定位为：

工程进度分析与汇报辅助系统。

它从“本地单机进度统计工具”升级为能够辅助用户发现问题、形成整改动作、持续跟踪闭环和生成汇报材料的本地系统。v2.0 仍优先服务单人本地使用，不急于做多人协作、权限体系或云端同步。

### 核心目标

1. 把 v1.9 已经能识别出的滞后项、预警记录和报表导出，进一步转化为可跟踪、可复查、可汇报的管理动作。
2. 在不破坏现有 Excel 导入、进度计算和报表导出稳定性的前提下，增加一个明确的新业务闭环。
3. 让系统从“分析结果展示”前进到“问题整改跟踪”。
4. 保持本地单机、portable、低部署成本的产品路线。

### 用户场景

- 监理、项目管理人员每周导入工程进度表，查看滞后项。
- 用户需要把滞后项拆成整改事项，记录责任人、责任单位和计划完成时间。
- 用户需要在下一次周报中说明上期整改项是否完成、是否关闭、是否超期。
- 用户需要导出整改跟踪表，便于现场会议、周例会和复查。
- 用户在本地长期保存多个项目和批次，不依赖服务器或多人在线协作。

### 主要痛点

- v1.9 可以导出整改清单，但整改项本身没有状态和复查记录。
- 滞后项每周都会出现，但缺少“从发现到关闭”的本地台账。
- Word 周报可以描述滞后，但不能沉淀整改闭环结果。
- 手工维护 Excel 整改台账容易和系统中的进度数据脱节。

### v2.0 不解决的问题

- 不做多人权限、审批流或在线协同。
- 不做云同步、移动端 App、Docker、PostgreSQL 或 Electron。
- 不重写 Excel 导入核心流程。
- 不改变进度计算公式。
- 不重写报表导出核心结构。
- 不把 AI 作为默认依赖。

### v2.0 与 v1.9 的区别

v1.9 解决“导入进度表后如何看、如何导出、如何本地保存”的问题。v2.0 建议解决“发现滞后后如何形成整改事项、如何跟踪、如何复查、如何带入汇报”的问题。

## 3. 候选方向评估

### 方向 A：AI 进度分析助手

价值：

- 可以让进度分析、周报说明、会议摘要和整改建议更自然。
- 对汇报材料质量有提升，尤其适合非技术用户快速整理文字。
- 可在现有规则化分析说明基础上增强，而不是替代。

复杂度：

- 中等。前后端配置、调用、错误处理和结果保存都可控。
- 难点不在技术调用，而在数据脱敏、提示词边界、用户信任和失败兜底。

风险：

- 数据隐私风险较高。工程进度表可能包含项目名称、责任单位、人员姓名和现场问题。
- AI 输出有不确定性，不能用于直接修改数据。
- 如果默认依赖 AI，会破坏本地单机稳定性。

建议优先级：

- 不建议作为 v2.0 主线。
- 建议作为 v2.1 或 v2.0 后续可选增强。
- 如果做，必须是本地可选功能，允许用户配置 API Base URL、API Key、Model，并保留规则化分析作为 fallback。

边界建议：

- 没有 API Key 时系统完全可用。
- 默认继续使用规则化分析。
- AI 只生成文字，不修改进度数据、整改状态或报表记录。
- AI 输出必须可编辑，并标注“AI辅助生成”。
- AI 调用失败不影响看板、报表和整改闭环。

### 方向 B：计划基线增强

价值：

- 对进度偏差准确性有直接价值。
- 多版基线、原始基线、当前基线和调整基线能解释“为什么计划进度变化”。
- 报表显示采用的基线，可以提升周报可信度。

复杂度：

- 中等偏高。需要明确基线数据结构、批次绑定关系、变更记录和报表展示方式。
- 与当前 Excel 导入关系密切，但不必修改导入核心逻辑，可以从基线管理页面和批次绑定开始。

风险：

- 如果定义不清，容易让用户混淆“Excel 内计划值”和“系统基线计划”。
- 报表和看板里计划进度来源必须解释清楚，否则会引入信任问题。

建议优先级：

- 适合作为 v2.0 的重要候选，但不建议压过整改闭环。
- 更适合作为 v2.1-baseline 或 v2.0-beta 后的增强项。

### 方向 C：导入能力增强

价值：

- 多 Sheet 批量导入是高频痛点，真实 Excel 往往按楼栋、专业或区域拆多个 Sheet。
- 导入前数据预处理、模板批量套用和导入历史对比也有实用价值。

复杂度：

- 中到高。多 Sheet 独立导入可控；同一 Sheet 多表格识别、合并单元格增强和智能表头识别复杂度高。
- 最容易破坏当前稳定导入流程。

风险：

- 当前 v1.9 导入流程已经稳定，贸然扩展会引入回归风险。
- 自动识别多个独立表格会带来大量边界情况。
- 不同 Sheet 是否合并、是否共用映射、失败是否回滚，都需要清晰产品规则。

建议优先级：

- 不建议作为 v2.0 主线，除非真实试用反馈显示导入复杂 Excel 是最大阻塞。
- 建议先做最小可行多 Sheet：多 Sheet 独立解析、独立映射、独立校验、独立生成 import_batch、支持批量发布。
- 不做跨 Sheet 自动合并，不做同 Sheet 多表格自动识别。

### 方向 D：整改闭环增强

价值：

- 非常适合监理和工程项目管理场景。
- 与 v1.9 的滞后项、预警记录、整改清单导出天然衔接。
- 比 AI 更实际，更能沉淀管理台账。
- 对本地单机版也有价值，用户可以在本地长期保存整改过程。

复杂度：

- 中等。需要新增数据表、列表页、编辑页、状态流转、导出和周报摘要。
- 不需要改 Excel 导入核心逻辑、字段识别、进度计算公式或现有报表导出核心，只需在现有结果之上建立整改项。

风险：

- 需要避免把本地单机系统做成复杂协同系统。
- 状态流转要简单，避免审批流、权限和多人分派。
- 要处理重复生成整改项的问题。

建议优先级：

- 建议作为 v2.0 主线。
- 最小版本只做本地整改台账，不做多人协作。

### 方向 E：图表和看板增强

价值：

- 专业 + 楼层交叉热力图、楼栋 + 专业矩阵、滞后分布图和趋势对比都适合工程进度。
- 对现场会议和管理层查看有帮助。

复杂度：

- 中等偏低到中等。多数图表可以基于现有 analytics 接口扩展。
- 大屏模式和自定义图表复杂度较高，容易变成 UI 工程。

风险：

- 容易做成“好看但管理闭环不足”。
- 自定义图表会带来配置保存、导出和兼容成本。

建议优先级：

- 可作为 v2.0 的辅助增强，但不建议作为主线。
- 最适合先做专业 + 楼层热力图、楼栋 + 专业矩阵、滞后分布图。

### 方向 F：本地数据管理增强

价值：

- 对本地长期使用非常重要。
- 备份管理、恢复向导、项目归档、批次冻结、数据库体检能降低长期使用风险。

复杂度：

- 中等。备份管理和恢复向导需要谨慎处理文件覆盖和运行中数据库。
- 数据归档、批次冻结涉及数据状态和误操作恢复。

风险：

- 恢复操作如果做成自动覆盖，风险较高。
- 需要清晰的“只读归档”和“可恢复备份”边界。

建议优先级：

- 建议作为 v2.0 的基础支撑能力，但不作为主线。
- 可在 v2.0-beta 或 rc 阶段补充备份列表、诊断增强和数据库体检。

## 4. 优先级评分表

评分说明：用户价值、复杂度、风险、架构影响和 v2.0 推荐度均为 1-5 分。复杂度、风险、架构影响分数越高表示越复杂、越高风险、影响越大。

| 方向 | 用户价值 | 复杂度 | 风险 | 架构影响 | v2.0 推荐度 | 结论 |

## 5. 推荐主线

推荐 v2.0 主线：

`v2.0-rectification`

重点做整改闭环。

### 为什么选择整改闭环

1. 它最顺畅地承接 v1.9 已完成的能力。v1.9 已经能识别滞后项、生成预警记录、导出整改清单，但缺少整改事项的状态和复查。
2. 它最贴合监理和工程管理真实工作流。用户发现滞后后，下一步不是继续看图，而是形成责任、期限、措施和复查。
3. 它不需要改变导入、字段识别、进度计算和报表导出核心逻辑，风险可控。
4. 它对本地单机版仍然有明确价值，不依赖多人协作。
5. 它能让 Word 周报和整改清单更像真实管理材料，而不是一次性导出文件。

### 为什么不先做其他方向

- 不先做 AI：AI 有文字价值，但隐私和输出不确定性高，不应成为本地单机正式主线的第一步。
- 不先做导入增强：多 Sheet 和复杂 Excel 很有价值，但会触碰当前最关键的稳定流程，建议等真实样本沉淀后单独做 `v2.0-import-plus` 或 `v2.1-import-plus`。
- 不先做计划基线：基线增强对准确性很重要，但它更偏计划管理体系，需要更仔细定义计划来源和报表解释，可以作为 v2.1 主线。
- 不先做图表增强：图表能提升展示，但不一定形成管理闭环。
- 不先做本地数据管理：应作为支撑能力逐步补强，但单独作为主线不如整改闭环直击业务价值。

## 6. v2.0 最小可行范围

v2.0 最小可行范围建议：

- 新增整改项数据表 `rectification_item`
- 新增整改项列表页
- 支持从滞后项生成整改项
- 支持从预警记录生成整改项
- 支持编辑责任人、责任单位、计划完成时间、备注
- 支持状态流转：open、in_progress、completed、closed、ignored
- 支持复查结果和关闭时间
- 支持导出整改跟踪表
- Word 周报增加整改摘要
- 系统维护或项目详情中展示整改项数量概览

最小版本不追求复杂流程，只要能让用户把“滞后项”变成“可跟踪整改项”即可。

## 7. v2.0 不做范围

v2.0 不做：

- 多人权限、责任人账号体系、审批流
- 云同步、移动端 App、WebSocket 通知
- PostgreSQL、Docker、Electron
- AI 默认集成或强依赖
- 自动改写 Excel 导入逻辑
- 自动识别同一 Sheet 多个表格
- 多 Sheet 自动合并为一个批次
- 改进度计算公式
- 重写报表导出核心逻辑
- 复杂看板自定义和大屏模式

## 8. 版本路线

### v2.0-alpha

目标：

最小可用版本，打通整改项从生成到关闭的主流程。

内容：

- 新增 `rectification_item` 数据表和基础 schema
- 新增整改项 CRUD API
- 新增整改项列表页
- 从滞后项生成整改项
- 从预警记录生成整改项
- 编辑责任人、责任单位、计划完成时间、备注
- 状态流转：open、in_progress、completed、closed、ignored

验收标准：

- 可以基于一个已发布批次生成整改项
- 同一进度项 / 预警记录不会重复生成相同整改项
- 可以编辑整改项责任信息
- 可以完成、关闭或忽略整改项
- 不影响现有导入、看板、预警和报表导出

### v2.0-beta

目标：

真实数据试用版本，验证整改闭环是否适合现场管理。

内容：

- 整改项筛选：状态、专业、楼栋、楼层、责任人、超期
- 整改项详情和复查记录
- 整改跟踪表导出
- Word 周报增加整改摘要
- Dashboard 或预警页提供生成整改项入口
- 备份包含整改项数据

验收标准：

- 能用真实项目数据完成“发现滞后 - 生成整改 - 修改责任 - 标记完成 - 复查关闭”
- 整改跟踪表可打开且列完整
- Word 周报能展示整改摘要
- 备份恢复后整改项不丢失
- 无 P0 / P1 问题

### v2.0-rc

目标：

候选发布版，补齐边界、文档、稳定性和回归测试。

内容：

- 完成整改闭环文档
- 完成接口和页面错误处理
- 完成重复生成、批次删除、项目删除相关边界处理
- 增加整改项相关测试
- portable 包验证
- 冒烟脚本加入整改闭环检查

验收标准：

- pytest 通过
- npm run build 通过
- portable 包可启动、备份、诊断、停止
- 整改闭环主流程通过
- 现有 v1.9 功能无回归

### v2.0-release

目标：

稳定版，可以作为本地工程整改闭环版本长期使用。

内容：

- 文档定稿
- release 包重新生成
- 最终冒烟验证
- 备份恢复验证
- 发布报告和标签准备

验收标准：

- 无 P0 / P1 阻塞问题
- 整改闭环真实数据试用通过
- release 包目录干净
- 报表、备份、诊断和维护页均正常
- 建议打标签 `v2.0-rectification-release`

## 9. 数据模型草案

推荐新增表：

`rectification_item`

字段：

- `id`
- `project_id`
- `batch_id`
- `progress_item_id`
- `warning_record_id`
- `discipline`
- `building`
- `floor`
- `system_name`
- `task_name`
- `issue_description`
- `delay_level`
- `responsible_person`
- `responsible_unit`
- `planned_finish_date`
- `status`
- `review_result`
- `remark`
- `created_at`
- `updated_at`
- `closed_at`

状态：

- `open`：未开始
- `in_progress`：整改中
- `completed`：已完成，待复查
- `closed`：已关闭
- `ignored`：已忽略

设计原则：

- 整改项保存现场定位字段快照，即使后续批次变化，历史整改记录仍可读。
- `progress_item_id` 和 `warning_record_id` 可为空，但至少应关联其中之一或保留手动创建来源。
- 重复生成应基于 `project_id + batch_id + progress_item_id + warning_record_id` 做去重。
- 关闭后不建议物理删除，只做状态变更。

## 10. 接口草案

接口草案：

- `GET /api/projects/{project_id}/rectifications`
- `POST /api/projects/{project_id}/rectifications`
- `POST /api/projects/{project_id}/rectifications/from-progress-items`
- `POST /api/projects/{project_id}/rectifications/from-warnings`
- `GET /api/rectifications/{id}`
- `PATCH /api/rectifications/{id}`
- `POST /api/rectifications/{id}/status`
- `POST /api/rectifications/{id}/close`
- `GET /api/projects/{project_id}/rectifications/export`
- `GET /api/projects/{project_id}/rectifications/summary`

页面草案：

- `/projects/:projectId/rectifications`：整改项列表
- `/projects/:projectId/warnings`：增加生成整改项入口
- `/projects/:projectId/dashboard`：滞后项区域增加生成整改项入口
- `/reports`：增加整改跟踪表导出记录

## 11. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 本地单机系统被做复杂 | 引入审批、权限、多人分派 | v2.0 明确只做本地台账和状态流转 |
| 重复生成整改项 | 同一滞后项反复生成多条 | 做去重策略和生成结果提示 |
| 状态含义不清 | 用户不知道 completed 和 closed 差别 | 页面文案说明：completed 为已完成待复查，closed 为复查关闭 |
| 与现有整改清单导出混淆 | 用户不知道导出清单和整改项台账区别 | 文档说明：整改清单是一次性导出，整改项是可跟踪台账 |
| 批次或项目删除影响整改项 | 删除测试数据导致孤儿记录 | 删除项目时级联处理；清理批次时提示关联整改项 |
| 周报内容过重 | Word 周报变长 | 周报只放摘要，详细内容放整改跟踪表 |
| 后续 AI 范围膨胀 | AI 想直接改整改建议或状态 | AI 只作为后续可选文字助手，不进入 v2.0 主线 |

## 12. 验收标准

v2.0 总体验收标准：

- 不破坏 v1.9 的 Excel 导入、字段识别、进度计算、看板、预警、报表导出、备份和 portable 包。
- 整改项可从滞后项和预警记录生成。
- 整改项可编辑责任信息、计划完成时间、复查结果和备注。
- 整改项状态流转清晰，支持 open、in_progress、completed、closed、ignored。
- 整改项可按项目、批次、状态、专业、楼栋、楼层筛选。
- 整改跟踪表可导出并打开。
- Word 周报可展示整改摘要。
- 备份恢复后整改项数据仍存在。
- pytest 通过。
- npm run build 通过。
- portable 包可重新生成并通过启动、诊断、备份、停止验证。

## 附：AI 方向边界预案

如果后续版本引入 AI，必须遵守以下边界：

1. AI 功能必须可选。
2. 没有 API Key 时系统仍然可用。
3. 默认继续使用规则化分析。
4. AI 只生成文字，不修改数据。
5. AI 输出必须可编辑。
6. AI 生成结果必须标注“AI辅助生成”。
7. 用户可以复制到 Word 周报。
8. 需要避免上传敏感数据，或明确提示用户。
9. 支持配置 API Base URL、API Key、Model。
10. 支持测试连接。
11. AI 调用失败时不影响系统。

## 附：导入增强最小方案预案

如果后续版本选择 `v2.0-import-plus` 或 `v2.1-import-plus`，建议多 Sheet 批量导入最小方案为：

1. 上传 Excel 后显示 Sheet 列表。
2. 用户可勾选多个 Sheet。
3. 每个 Sheet 独立解析。
4. 每个 Sheet 独立字段映射。
5. 每个 Sheet 独立校验。
6. 每个 Sheet 生成独立 import_batch。
7. 支持批量发布。
8. 不做跨 Sheet 强制合并。
9. 失败的 Sheet 不影响成功的 Sheet。
10. 最终给出批量导入结果汇总。

明确不做：

- 自动识别同一 Sheet 多个表格。
- 多 Sheet 自动合并为一个批次。
- 不同 Sheet 共用一套映射并强制套用。
